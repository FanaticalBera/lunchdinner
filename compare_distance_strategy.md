# LunchDinner 거리 정보 보조화 메모

> 작성일: 2026-03-06
> 목적: 지도 없이 `distance` 기준을 더 설득력 있게 다루기 위한 설계 메모

## 1. 문제 정의

현재 `distance`는 compare 기준에 포함되어 있지만, 실제 거리 데이터와 연결되어 있지는 않다.

지금 상태의 의미는 사실상 아래 둘 중 하나다.

- 사용자가 머릿속으로 체감 거리를 점수화
- 다른 기준과 동일한 방식으로 임의 평가

이 구조는 가능은 하지만, 사용자가 `distance`를 어떻게 판단해야 하는지 애매해질 수 있다.

## 2. 목표

- 지도 API 없이도 거리 기준을 다룰 수 있어야 한다.
- 사용자가 복잡한 판단 없이 입력할 수 있어야 한다.
- 나중에 실제 위치 정보가 들어와도 확장 가능해야 한다.

## 3. 1차 권장안: 체감 거리 3단계 입력

### 개념

후보별 거리 평가를 아래 3단계로 단순화한다.

- 가깝다
- 보통
- 멀다

### 장점

- 사용자가 훨씬 빨리 판단할 수 있다.
- 지도 권한/외부 API가 필요 없다.
- 현재 점수 구조와도 쉽게 연결 가능하다.

### 현재 구조와 결합 방식

예시 매핑:

- 가깝다 -> 5점
- 보통 -> 3점
- 멀다 -> 1점

또는 가중치 반영 시 별도 enum으로 유지한 뒤 계산 직전에 점수화 가능하다.

## 4. 2차 권장안: 시간 기준 문구 사용

`distance` 대신 내부 문구를 더 자연스럽게 바꾸는 방법도 있다.

예:

- 가까움
- 이동 편의
- 가기 편함

특히 compare는 실제 직선거리보다 "지금 가기 귀찮지 않은가"가 더 중요할 수 있다.

즉, UI 문구는 `거리`보다 `이동 편의`가 더 맞을 가능성이 있다.

## 5. 3차 확장안: 예상 이동 시간 보조정보

나중에 실제 위치 정보가 필요해지면 전체 지도를 여는 대신 아래만 보조 정보로 붙이는 것이 좋다.

예:

- 도보 8분
- 이동 12분 예상
- 현재 위치 기준 비교적 가까움

이 방식은 지도 없이도 결과 설명력을 올릴 수 있다.

## 6. 비추천 방향

현재 단계에서 아래는 권장하지 않는다.

- Kakao Map API/네이버 지도 API를 바로 메인 기능으로 도입
- compare 화면에 지도 전체를 띄우는 방식
- 후보 탐색과 compare를 한 화면에서 동시에 해결하려는 방식

이건 compare UX 개선보다 훨씬 큰 범위다.

## 7. 확장 포인트

나중에 위치 기반 데이터를 붙일 수 있도록 아래 구조를 염두에 둘 수 있다.

```ts
interface CandidateDistanceMeta {
  distanceMode: 'manual' | 'estimated';
  distanceLevel?: 'near' | 'medium' | 'far';
  estimatedMinutes?: number;
}
```

초기에는 `distanceLevel`만 쓰고, 추후 `estimatedMinutes`를 붙이는 방식이 가장 안전하다.

## 8. 추천 결론

현재 시점의 추천은 아래와 같다.

1. `distance`를 그대로 두기보다 `이동 편의` 성격으로 해석을 명확히 한다.
2. 지도 없이 3단계 입력 또는 단순 점수 입력으로 먼저 정리한다.
3. 이후 필요가 생기면 예상 이동 시간만 보조 정보로 붙인다.
4. 지도 API 도입은 마지막 단계에서 검토한다.

즉, 지금의 정답은 `지도`가 아니라 `거리 기준의 단순화`다.
