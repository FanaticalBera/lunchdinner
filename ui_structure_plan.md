# 🍱 "대신 골라줘!" UI 골격 및 화면 명세서 (Screen Structure Document)

## A. 전체 화면 맵 (Screen Map)

사용자가 목표에 도달하기까지 거치는 핵심 5단계 화면과 레이어(모달)의 흐름입니다.

1. **[Screen 1] Intro & Mode Select**: 점심/저녁 상황 선택 (진입점)
2. **[Screen 2] Weight Wizard**: 4가지 기본 기준의 가중치(중요도) 슬라이더 조절
   - └ *[Modal]* 기준 추가/변경 모달 ( MVP 이후 확장성 고려 )
3. **[Screen 3] Candidate Input**: 비교할 식당/메뉴 후보 태그 입력
4. **[Screen 4] Scoring Board**: 기준별 후보 식당 점수 평가 탭
5. **[Screen 5] Result & Justification**: 1위 결과 발표 및 컴포넌트 차트 뷰

**이동 흐름**: `[1] ➔ [2] ➔ [3] ➔ [4] ➔ [5]` (선형 진행이 원칙이며 상단 `< 뒤로가기`를 통해 직전 상태로 복귀 가능)

## B. 단계별 상세 화면 명세

### [Screen 1] Intro & Mode Select
- **화면 목적**: 서비스 진입 및 분위기(테마) 설정
- **필요 요소 (입력/버튼)**: 
  - [점심 먹을래 🌞] 거대 버튼
  - [저녁 먹을래 🌙] 거대 버튼
- **버튼 활성화 조건**: 상시 활성화
- **다음 단계 이동 조건**: 두 버튼 중 하나 탭 시 즉시 Screen 2로 이동
- **복귀 시 유지 값**: 없음 (초기화면)
- **UX 장치**: 선택 요소가 딱 2개뿐이라 오작동이나 고민의 여지를 원천 차단.

### [Screen 2] Weight Wizard (가중치 설정)
- **화면 목적**: 항목별 중요도 100% 분배
- **필요 요소**:
  - 4개의 범위 슬라이더 (Range Slider): 맛, 가성비, 거리, 대기시간
  - 남은 가중치 표시 바 (Visual Indicator)
  - [다음 ➔] 플로팅 액션 버튼(FAB) 또는 하단 고정 버튼
- **버튼 활성화 조건**: 남은 가중치가 0%이거나 할당 에러 상태가 아닐 때 (기본값으로 25%씩 맞춰져 있어 진입 시점부터 활성화 상태)
- **다음 단계 이동 조건**: [다음] 버튼 탭
- **복귀 시 유지 값**: 직전에 맞춘 퍼센티지 값 100% 유지
- **에러/예외 상태**: 사용자가 하나의 슬라이더를 100%로 끝까지 당겨, 다른 3개가 강제로 0%가 되었을 때의 한계점 도달 상태 (슬라이더 Lock 걸림)
- **UX 장치**: 사용자가 조작을 포기하고 바로 [다음]을 눌러도 작동하도록 **초기 균등 분배(각 25%) 상태를 디폴트로 제공**.

### [Screen 3] Candidate Input (후보 입력)
- **화면 목적**: 비교 대상 수집
- **필요 요소**:
  - 후보명 입력 Text Input 필드 (단일 폼)
  - 등록된 후보가 쌓이는 Chip(태그) 컨테이너
  - 활성화/비활성화되는 [점수 매기기 ➔] 버튼
- **버튼 활성화 조건**: Chip(후보)이 **2개 이상** 등록되었을 때 활성화
- **다음 단계 이동 조건**: 활성화된 버튼 탭
- **복귀 시 유지 값**: 등록해 둔 Chip 리스트 전체 유지
- **에러/예외 상태**: 
  - 이미 등록된 이름과 똑같은 텍스트 엔터 시 ➔ `[중복 에러 피드백]`
  - 내용 없이 엔터 시 ➔ `[무시]`
- **UX 장치**: 인풋 창 포커스 시 모바일 키보드가 올라오며, 키보드 바로 위에 찰싹 붙는 형태의 [입력/추가] 버튼 배치. 

### [Screen 4] Scoring Board (점수 평가 보드)
- **화면 목적**: 빠르고 직관적인 기초 데이터 마킹
- **필요 요소**:
  - 상단 탭 네비게이션: `[ 맛 ]`, `[ 가성비 ]`, `[ 거리 ]`, `[ 대기시간 ]`
  - 리스트 뷰: (등록한 후보 수만큼 렌더링 된 Row)
  - 평가 컨트롤: 각 Row마다 1~5단계 [😍 🙂 😐 😕 😡] 이모지 라디오 버튼 세트
  - 최하단 [🔥 결과 보기] 버튼
- **버튼 활성화 조건**: 4개의 탭을 적어도 **한 번씩 다 눌러서 확인했을 때** 활성화 (사용자가 디폴트 3점 상태를 의도적으로 패스했음을 시스템이 인지하기 위함)
- **다음 단계 이동 조건**: [결과 보기] 버튼 탭
- **복귀 시 유지 값**: 각 항목별로 찍어둔 1~5점 점수 매트릭스 전체 유지
- **UX 장치**: 화면 진입 시 모든 후보의 모든 항목 점수가 **"3점(😐)"으로 디폴트 세팅**되어 있음. 사용자는 특별히 좋고 나쁜 항목만 툭툭 치고 넘길 수 있음.

### [Screen 5] Result & Justification (결과)
- **화면 목적**: 결과 수치화 및 감성적 납득
- **필요 요소**:
  - 최상단 1위 카드 뷰 (이미지/아이콘 + 이름)
  - 1위 선정 이유 텍스트 블록
  - 1~3위 누적 막대 차트 (Stacked Bar Chart: 각 항목 점수가 색상별로 구분된 바)
  - [가중치만 수정하기] 보조 버튼
  - [처음부터 새로하기] 메인 버튼
- **버튼 활성화 조건**: 상시 활성화
- **UX 장치**: 결과 화면 로딩 전 1~2초간 스피너나 프로그래스 애니메이션을 두어 '열심히 계산하는 느낌(Labor Illusion)' 부여.

## C. 텍스트 와이어프레임 (Text Wireframe)

**[Screen 2] 가중치 설정 화면**
```text
상단 : [ < 뒤로 ]            [==        ] 1/4 Step
       "이번 식사에서 제일 중요한 건 뭐예요?"
       (조절하지 않아도 바로 다음으로 넘길 수 있어요)
--------------------------------------------------
중단 : 
       [ 맛 ]              [===========O====] 40%
       
       [ 가성비 (가격) ]    [=====O==========] 20%
       
       [ 가까움 단계 ]      [=====O==========] 20%
       
       [ 빠른 대기시간 ]    [=====O==========] 20%

       (💡 남은 비중: 0% / 100%가 채워졌어요!)

                                   (활성화 상태)
```

```text

--------------------------------------------------
중단 : 
      👉[  맛  ]👈   [ 가성비 ]   [ 거리 ]   [ 대기 ]

      🥘 김치찌개
      😡 매우별로  😕  [😐 보통]  🙂  😍 최고야

      😡 매우별로  😕  😐 보통  [🙂]  😍 최고야

--------------------------------------------------
하단 :                            [ 🔒 결과 보기 ]
                        (탭을 끝까지 보면 열려요!)
```

## D. 상태 흐름 (UI State Flow)

프론트엔드 상태(State) 관리 관점에서의 필수 뼈대입니다. 이 데이터가 유지되어야 화면 간 앞뒤 이동이 안 깨집니다.

1. **`currentStep`**: `number` (1~5)
   - 역할: 현재 렌더링해야 할 화면 컴포넌트를 결정. (<뒤로>, <다음> 버튼이 증감시킴)
2. **`lunchOrDinner`**: `'lunch' | 'dinner' | null`
   - 생성점: Screen 1. 이후 유지. 전체 테마 CSS 클래스를 결정함.
3. **`criteriaWeights`**: `Object { taste: 40, price: 20, distance: 20, time: 20 }`
   - 생성점: Screen 2. 진입 시 디폴트 세팅. 슬라이더로 변경 시 항상 썸(Sum)=100을 맞추도록 로직 작동.
4. **`candidates`**: `Array [ { id: 'c1', name: '김치찌개' }, { id: 'c2', name: '돈까스' } ]`
   - 생성점: Screen 3. 엔터 칠 때마다 배열 꼬리에 추가, [x] 누를 때 배열에서 삭제됨.
5. **`scores`**: `Object { c1: { taste: 3, price:3... }, c2: { taste: 4, price:3... } }`
   - 생성점: Screen 3에서 <다음> 누르고 4 진입 시 데이터 골격 자동 생성. 초기값 무조건 `3`.
   - 수정점: Screen 4 평가 보드.
6. **`viewedTabs`**: `Array [ 'taste', 'price'... ]`
   - 역할: Screen 4에서 유저가 4개의 탭을 한 번씩 다 구경했는지 체크하는 불리언 플래그 배열. 모두 찍히면 결과 버튼 `disabled = false`.

## E. 인터랙션 뼈대 (Interaction Skeleton)

1차 MVP 개발에 적합한 가볍고 명확한 시각적 피드백 위주의 연출.

* **슬라이더 조작**: '맛' 슬라이더 썸(동그라미)을 오른쪽으로 끌고 갈 때, 아래의 '가성비', '거리' 슬라이더의 썸이 실시간으로 찌그러지듯 왼쪽으로 자동 후진하는 애니메이션 연동.
* **후보 추가/삭제**: 엔터를 치면 폼 텍스트 창이 비워지는 동시에, 아래 빈 공간에 태그 칩이 *Pop-in(크기가 0에서 1로 커지는 탄성 효과)* 됨. [x] 터치 시 *Fade-out*되며 삭제.
* **점수 스와이프**: [맛] 탭을 다 찍고 [가성비] 탭을 터치하면, 리스트 뷰 전체가 왼쪽으로 밀려나가고 오른쪽에서 새 리스트가 들어오는 *슬라이드 트랜지션*.
* **결과 화면 진입**: <결과 보기> 탭 ➔ 화면이 살짝 어두워지며 중앙에 '계산기 아이콘이 돌아가는 로딩 뷰 (1.5초)' ➔ 축하 폭죽(Confetti 효과)이 화면 상단에서 흩뿌려지며 1위 카드 *Scale-Up 등장*.

## F. 검증 규칙 (UI Validation Requirements)

서버가 없어도(사용 안 해도) 브라우저 앞단에서 막고 경고해야 하는 필수 룰.

1. **가중치 슬라이더 한계선**: 
   - 규칙1: 하나의 슬라이더는 최대 100%을 넘지 못함. 
   - 규칙2: 합은 반드시 100%. A를 늘리면 B, C, D에서 1/3씩 삭감. B,C,D가 모두 0이면 A는 더 이상 우측 드래그 안 됨.
2. **후보명 Validator**:
   - 최소 길이 1자 이상, 최대 15자 제한 (레이아웃 깨짐 방지).
   - 앞뒤 공백(Trim) 제거 후 검사.
   - `candidates` 배열 내 동일 문자열 일치 시 "이미 있어요!" Toast 알림 및 입력창 값 유지.
3. **가짜 진행 방지 (Score Board)**:
   - 디폴트 3점이라고 해서 들어오자마자 [결과보기]를 누르는 '실수(휴먼 에러)' 방어.
   - 탭 네비게이션을 전부 순회한(Viewed) 흔적이 State에 남아야만 하단 버튼 언락.

## G. MVP 화면 범위 (Scope)

**🟢 1차 버전에 반드시 포함 (Must-Have UI)**
- Screen 1 ~ Screen 5 뼈대 컴포넌트 전체.
- 상단 공통 Header (진행 상태 바, 뒤로가기 버튼).
- 테마 분기 로직 (점심: Light mode base, 저녁: Dark mode base).
- 하드코딩된 기본 4개 기준 (맛, 가성비, 접근성, 대기시간).

**🟡 지금은 제외하는 요소 (Out of Scope for v1)**
- 기준 '추가/삭제' 모달 (4개 고정으로 시작하여 가중치 계산 로직의 안정성 1순위 타겟).
- 후보 명칭의 자동완성 / 지도 연동 드롭다운.
- 로그인 / 결과 공유하기(카카오톡 링크 등) / 히스토리 내역.

## H. 디자인/프론트엔드 작업 전 5대 체크리스트

디자이너/개발자가 코드 및 피그마를 열기 전 "YES"가 나와야 합니다.

1. [ ] **Time to Task**: 유저가 앱을 켜고 결과를 보기까지 아무 고민 없이 '다음'만 연타했을 때 **30초** 안에 1위 결과가 나오는가? 
2. [ ] **Fatigue Test**: 5개의 식당을 다 평가하는데 손가락 터치가 쾌적하게 한 방향(아래/위)으로만 스크롤 되며 평가되는가?
3. [ ] **Label Clarity**: "가격 지표가 높아야 싼 거다" 라는 것을 바보도 알 수 있게 "가성비 만족도", "싸고 양 많음 선호도" 등으로 텍스트 라벨링이 확실하게 박혀 있는가?
4. [ ] **Responsive**: 후보명을 "엄청나게긴이름의무언가치킨전문점" 이라고 쳤을 때 태그나 리스트 뷰가 화면 밖으로 튀어나가 깨지지 않는가?
5. [ ] **Justification**: 사용자가 1위 결과를 보고 "아하, 내가 아까 싼 걸 40%나 중요하다고 했지!" 라고 바로 인정할 수 있는 시각 정보가 결과창에 명확한가?
