# LunchDinner 비교하기 카테고리 검토 보고서 개정판

> 작성일: 2026-03-06
> 개정 기준: 기존 보고서 + 추가 AI 리뷰 + 재검토 의견 반영
> 범위: `compare` 플로우 중심 UX/기능 검토
> 기준 코드: `src/App.tsx`, `src/screens/CandidateInputScreen.tsx`, `src/screens/ScoringBoardScreen.tsx`, `src/screens/ResultScreen.tsx`, `src/domain/recommendation.ts`, `src/domain/types.ts`

## 1. 현재 상태 요약

현재 `compare` 플로우는 아래 4단계로 구성되어 있다.

1. 가중치 설정
2. 후보 입력
3. 점수 평가
4. 결과 확인

기술적으로는 프로토타입 수준에서 충분히 작동한다.

- 상태는 reducer로 통합되어 화면 이동 중에도 유지된다.
- 점수 계산은 순수 함수로 분리되어 있다.
- 입력 누락 시 결과 화면 진입을 막는 가드도 있다.
- 동점 처리와 기본 예외 상황도 최소한 방어되어 있다.

즉, 현재 상태는 여전히 "작동은 하는 compare"다. 다만 문제는 기능 부재가 아니라, 실제 사용 시 결정 경험이 아직 무겁고 설득력이 약하다는 점이다.

## 2. 핵심 진단

### 2-1. 결과는 나오지만 결과를 믿게 만들지는 못한다

현재 결과 화면은 아래 수준에서 멈춰 있다.

- 1위 후보
- 종합 점수
- 순위
- 한 줄 요약
- 상위 3개 후보 점수 목록

이 구조에서는 사용자가 "왜 이 후보가 이겼는지"를 이해하기 어렵다.

핵심 문제는 UI만 단순한 것이 아니라, 도메인 데이터 자체가 얇다는 점이다.

현재 `Result` 타입과 `buildCompareResult()`는 총점 중심 구조다. 그래서 기준별 기여도, 1위와 2위 차이, 항목별 우열 같은 설명 데이터를 화면에 전달할 수 없다. 즉, 결과 화면 개선은 단순 UI 수정이 아니라 **도메인 타입 확장 -> 계산 함수 확장 -> UI 반영** 순서로 접근해야 한다.

### 2-2. compare는 입력 피로도가 높다

현재 점수 평가는 `맛 / 가성비 / 거리` 탭을 오가며 후보마다 1~5점 이모지 버튼을 눌러야 한다.

- 후보 3개면 최소 9회 입력
- 후보 4개면 최소 12회 입력
- 탭 이동과 판단까지 포함하면 체감 피로도는 더 큼

여기서 진짜 문제는 클릭 수 자체보다, 사용자가 계속 멈춰서 판단해야 한다는 점이다.

### 2-3. 인지 부하가 점수 입력에만 있는 것이 아니다

기존 보고서보다 더 분명히 짚어야 할 부분이 있다. compare가 무거운 이유는 `점수 평가`만이 아니라, 그 전에 이미 `가중치 설정`도 사용자에게 추상적 판단을 요구한다는 점이다.

즉 사용자는 compare 과정에서 두 번 생각해야 한다.

- 무엇이 중요한가: 가중치 설정
- 각 후보가 얼마나 좋은가: 점수 평가

이 이중 판단 구조가 compare 플로우 전체를 무겁게 만든다.

### 2-4. 평가 기준의 모호성은 낮은 우선순위가 아니라 P2 내부 문제다

`맛 / 가성비 / 거리`라는 기준 자체는 쓸 만하다. 다만 현재는 각 기준을 어떤 의미로 판단해야 하는지 안내가 부족하다.

예:

- 맛: 객관적 맛인지, 오늘 당기는 정도인지 애매함
- 가성비: 사람마다 예산 기준이 다름
- 거리: 실제 거리인지 체감 거리인지 모호함

이 문제는 별도 대주제로 분리할 정도는 아닐 수 있지만, 점수평가 UX를 손볼 때 반드시 함께 해결해야 한다. 즉, 우선순위가 낮은 이슈라기보다 **P2 안에 포함되어야 할 이슈**다.

### 2-5. 후보 수 제한 부재는 작지만 중요한 구조적 문제다

현재 후보 입력에는 상한이 없다. 이 상태에서는 사용자가 5개, 6개 이상도 넣을 수 있고, 그러면 compare 플로우 전체가 급격히 무거워진다.

이건 단순 입력 편의성 문제가 아니라 compare UX 전체의 질을 좌우하는 요소다. 따라서 후보 입력 품질 개선 전체는 후순위로 두더라도, **후보 수 상한 제한만큼은 앞당겨 처리할 가치가 있다.**

## 3. 지도 기능에 대한 판단

### 결론

현 단계에서는 지도 도입을 메인 범위로 넣지 않는 것이 맞다.

### 이유

지도는 단순한 부가 기능이 아니다. 지도를 넣는 순간 앱의 성격이 바뀐다.

- 현재 앱 성격: 결정 보조 앱
- 지도 도입 후 성격: 주변 식당 탐색 앱

이 전환이 일어나면 아래 작업이 함께 따라온다.

- 위치 권한 처리
- 지도 API 연동
- 주변 식당 데이터 품질 문제
- 거리 계산 신뢰성
- 폐업/중복/누락 데이터 처리
- 결과 없음 상태 UX

즉, `distance`를 보강하는 문제와 `지도 서비스화`는 같은 레벨의 문제가 아니다.

### 현실적인 방향

지금 필요한 것은 지도 자체가 아니라 `거리 기준을 어떻게 가볍게 반영할 것인가`다.

현실적인 축소안:

- 후보별 `가깝다 / 보통 / 멀다` 수동 선택
- 추후 현재 위치 기준 예상 거리만 보조 정보로 추가
- 결과 화면에서 거리 요약만 보여주고 지도 전체는 열지 않음

정리하면 위치 기반 필터링은 좋은 확장 아이디어지만, 지금 시점의 1순위는 아니다. 우선순위상으로는 compare 핵심 UX 정리 이후 검토가 맞다.

## 4. 점수평가 방식에 대한 판단

### 결론

현재 방식은 프로토타입으로는 가능하지만, 장기적으로는 유지하기 어려운 UX다.

### 이유

현재 방식은 사용자가 다음을 모두 해야 한다.

- 평가 기준을 이해해야 함
- 가중치를 먼저 정해야 함
- 후보를 기억해야 함
- 탭을 이동해야 함
- 각 후보의 절대점수를 정해야 함

이 흐름은 "빨리 점심/저녁을 정하고 싶다"는 앱 목적과 잘 맞지 않는다.

## 5. 대안 검토

### 대안 A. 기준별 승자 선택 방식

예:

- 맛은 어디가 더 끌리나요?
- 가성비는 어디가 더 나은가요?
- 가까운 곳은 어디인가요?

장점:

- 입력 횟수가 크게 줄어든다.
- 절대점수 대신 직관적 선택이 가능하다.
- compare의 목적과 잘 맞는다.

주의점:

- 단순 UI 교체가 아니라 계산 로직 재설계가 필요하다.
- 현재의 `절대점수 x 가중치` 구조를 `기준별 승리 x 가중치` 또는 유사 구조로 다시 설계해야 한다.

즉, 사용자 경험 측면에서는 가장 유력한 방향이지만, 구현 난이도는 가볍지 않다.

### 대안 B. 상대비교 방식

예:

- A와 B 중 더 맛있을 것 같은 곳은?
- A와 B 중 더 부담 없는 곳은?

장점:

- 사용자가 점수 기준을 세우지 않아도 된다.
- 판단이 자연스럽다.

단점:

- 후보 수가 늘어나면 비교 조합이 빠르게 증가한다.
- 후보 수 상한이 없으면 구조적으로 커버하기 어렵다.

따라서 이 방식은 후보 수를 강하게 제한할 수 있을 때만 의미가 있다.

### 대안 C. 현재 구조 유지 + 입력 보조 강화

현 구조를 당장 유지해야 한다면 최소한 아래는 필요하다.

- 탭별 완료 상태 표시
- 미입력 후보 강조
- 기준 설명 문구 추가
- 1~5점의 의미를 명확히 안내
- 후보 수 상한 제한

예:

- 맛 1점: 별로 안 당김
- 맛 5점: 오늘 가장 먹고 싶음

이 방식은 기존 구현을 덜 흔들면서 개선할 수 있지만, 근본적으로는 현재 구조를 연장하는 접근이다.

## 6. 권장 방향

지금 단계에서 가장 현실적인 권장 방향은 아래와 같다.

### 6-1. 결과 설명력을 먼저 확보한다

compare는 계산기보다 설득 도구에 가까워야 한다. 따라서 다음 개선의 첫 단추는 결과 화면의 디자인이 아니라 결과 데이터 구조 자체다.

반드시 추가 검토할 요소:

- 기준별 점수 breakdown
- 가중치 반영값
- 1위와 2위 차이
- 동점 사유
- 다음 액션 안내

### 6-2. 후보 수를 제한한다

후보 입력 품질 전체를 손보기 전에, 우선 compare 후보 수 상한을 명시하는 것이 좋다.

권장 예:

- 최대 4개까지만 입력 허용
- 권장 수는 2~3개로 안내

이 조치만으로도 이후 비교/평가/결과 단계의 무게가 크게 줄어든다.

### 6-3. 점수평가 UX는 축소 또는 대체 대상으로 본다

단기적으로는 입력 보조 강화가 가능하고, 중기적으로는 기준별 승자 선택 방식으로 전환하는 것이 가장 유력하다.

즉, 점수평가 화면은 개선 대상이 아니라 **재설계 대상**으로 보는 쪽이 맞다.

### 6-4. 지도는 보조 정보 후보로만 남긴다

지도 API 연동은 당장 착수 항목이 아니라, compare 핵심 UX를 정리한 후 검토할 수 있는 확장 카드로 관리한다.

## 7. 개정 우선순위

기존 보고서보다 구현 현실을 반영해 우선순위를 아래처럼 조정한다.

### P1. 결과 데이터 구조 확장

목표: 결과 설명력을 만들 수 있는 기반 확보

작업 예:

- `Result` 타입 확장
- `buildCompareResult()`에 기준별 데이터 추가
- ranking 항목 세분화

### P2. 결과 화면 개선

목표: 왜 1위인지 한눈에 보이게 만들기

작업 예:

- 기준별 breakdown UI 추가
- 1위 vs 2위 비교 영역 추가
- 동점/근소한 승부 문구 개선

### P2.5. 후보 수 상한 제한

목표: compare 플로우 전체 무게 제어

작업 예:

- 후보 최대 개수 제한
- 권장 후보 수 안내
- 상한 초과 시 안내 메시지 추가

### P3. 점수평가 UX 축소 또는 대체

목표: 인지 부하 줄이기

작업 예:

- 현 구조 유지 시 완료 상태/기준 설명/점수 의미 보강
- 중기적으로 질문형 또는 기준별 승자 선택 방식 검토

### P4. 후보 입력 품질 개선

목표: 입력 품질 및 편의성 보강

작업 예:

- 공백/표기 차이 정규화 중복 방지
- 후보 수정 기능 추가
- 예시 후보 칩 제공

### P5. 거리 정보 보조화

목표: 지도 없이도 거리 기준을 다룰 수 있게 만들기

작업 예:

- 후보별 `가깝다 / 보통 / 멀다` 선택
- 추후 예상 거리 보조 정보 연결 검토

## 8. 추가로 유지할 관점

향후 검토 항목으로 아래도 유지할 가치가 있다.

- 가중치 설정 단계 자체의 단순화 필요성
- 비교 결과 이력 또는 미니 히스토리 기능
- compare 결과의 재시도 경험 개선

이 항목들은 지금 당장 핵심은 아니지만, compare를 반복 사용하게 만드는 방향에서는 중요하다.

## 9. 최종 결론

현재 compare 카테고리의 문제는 "기능이 없다"가 아니다. 문제는 아래 두 가지다.

1. 결과가 충분히 설명되지 않는다.
2. 결정을 내리기까지 사용자가 너무 많이 생각해야 한다.

따라서 다음 작업 순서는 아래가 맞다.

1. 결과 데이터를 더 풍부하게 만든다.
2. 결과 화면을 설명형으로 바꾼다.
3. 후보 수를 제한해 compare의 무게를 줄인다.
4. 점수평가 방식을 축소하거나 재설계한다.
5. 거리 기준은 지도 없이도 다룰 수 있게 단순화한다.

정리하면:

- 지도: 지금은 메인 스코프에서 보류
- 결과 화면: UI보다 먼저 도메인 확장이 필요
- 후보 수 제한: 예상보다 중요한 선행 과제
- 점수평가: 단순 개선 대상이 아니라 재설계 후보
- compare의 다음 단계: "정확한 계산"보다 "자연스러운 결정" 중심으로 이동
