# LunchDinner Compare 재설계 탐색 메모

> 작성일: 2026-03-06
> 목적: 현재 `weights + scores` 기반 compare를 유지할지, 질문형/승자 선택형으로 전환할지 판단하기 위한 메모

## 1. 현재 구조 요약

현재 compare 상태 구조는 아래 흐름이다.

1. 가중치 설정
2. 후보 입력
3. 후보별 절대점수 입력
4. 가중합 결과 계산

핵심 데이터는 아래 두 축이다.

- `weights`: 사용자가 중요도를 직접 분배
- `scores`: 각 후보의 절대점수(1~5)

즉, 현재 구조는 `후보 평가`와 `기준 중요도`를 모두 사용자에게 직접 맡긴다.

## 2. 현재 구조의 장점

- 계산식이 명확하다.
- 도메인 로직이 이미 구현돼 있다.
- 결과 설명을 수치 기반으로 풀어내기 쉽다.
- 동점 처리나 순위 계산이 단순하다.

## 3. 현재 구조의 한계

- 사용자가 두 번 판단해야 한다.
  - 무엇이 중요한가
  - 각 후보가 얼마나 좋은가
- 절대점수 기준이 사람마다 흔들린다.
- 후보 수가 늘면 입력량이 급격히 커진다.
- compare의 목적이 "빨리 정하기"인데, 체감상 설문처럼 느껴질 수 있다.

## 4. 대안 구조 A: 기준별 승자 선택 방식

### 개념

기준마다 1등 후보만 고른다.

예:

- 맛은 어디?
- 가성비는 어디?
- 거리는 어디?

### 상태 구조 초안

```ts
interface CompareChoiceState {
  weights: Weights;
  candidates: Candidate[];
  winnersByCriterion: Partial<Record<CriterionKey, Candidate['id']>>;
}
```

### 계산 규칙 초안

- 각 기준의 승자 후보에게 해당 기준 가중치를 부여
- 기준 승리 수보다 `획득 가중치 총합`으로 결과 계산
- 동점이면 공동 1위 또는 추가 질문 유도

### 장점

- 입력 횟수가 크게 줄어든다.
- 사용자가 절대점수 기준을 세울 필요가 없다.
- 앱 목적과 더 잘 맞는다.

### 단점

- 현재 계산 로직을 그대로 재사용하기 어렵다.
- 한 기준 안에서 2등/3등 정보가 사라진다.
- 사용자가 "비슷한데 굳이 하나만 고르기 어렵다"고 느낄 수 있다.

## 5. 대안 구조 B: 질문형 상대비교 방식

### 개념

질문 단위로 두 후보 또는 여러 후보 중 더 나은 쪽을 고른다.

예:

- 지금 더 당기는 곳은?
- 지금 예산에 더 편한 곳은?
- 지금 이동하기 덜 귀찮은 곳은?

### 상태 구조 초안

```ts
interface CompareQuestionState {
  candidates: Candidate[];
  answers: Array<{
    criterion: CriterionKey;
    winnerCandidateId: Candidate['id'] | null;
  }>;
}
```

### 장점

- 질문 문장이 자연스럽다.
- 사용자가 점수를 숫자로 바꾸지 않아도 된다.

### 단점

- 후보 수가 많아지면 조합 설계가 복잡해진다.
- 현재 가중치 구조와 자연스럽게 합치는 추가 설계가 필요하다.

## 6. 현재 구조 유지안

### 개념

기존 `weights + scores` 구조는 유지하되 입력 보조를 강화한다.

이미 반영된 보강:

- 탭별 진행 상태
- 기준 설명 문구
- 1~5점 의미 안내
- 미입력 후보 강조
- 후보 수 상한

### 추가 보강 여지

- 한 기준을 모두 입력하면 자동으로 다음 탭 이동
- 아직 입력 안 된 후보를 상단으로 정렬
- 결과 계산 전 미입력 후보 목록 요약

### 장점

- 기존 도메인 로직을 그대로 활용할 수 있다.
- 구현 리스크가 가장 낮다.

### 단점

- 근본적으로는 여전히 절대점수 입력 구조다.

## 7. `절대점수 x 가중치` vs `기준별 승리 x 가중치`

| 항목 | 절대점수 x 가중치 | 기준별 승리 x 가중치 |
|------|------------------|----------------------|
| 사용자 입력량 | 많음 | 적음 |
| 계산 설명력 | 높음 | 중간 |
| 구현 난이도 | 현재 낮음 | 재설계 필요 |
| 직관성 | 중간 | 높음 |
| 세밀한 차이 표현 | 가능 | 제한적 |
| 제품 목적 적합성 | 보통 | 높음 |

## 8. UI 시안 수준 흐름 제안

### 질문형 compare 초안

1. 가중치 설정
2. 후보 입력
3. 기준별 승자 선택
4. 결과 화면

### 기준별 승자 선택 화면 초안

- 상단: 현재 기준 설명
- 본문: 후보 카드 2~4개
- 액션: 각 카드 클릭 시 해당 기준 승자 선택
- 하단: "이 기준은 비슷함" 보조 선택지 가능

## 9. 추천 결론

지금 당장 구조를 뒤집는 것은 권장하지 않는다.

추천 순서:

1. 현재 구조 보강 완료 상태를 한 번 써본다.
2. 실제 사용 피드백에서 여전히 점수 입력이 무겁다면 전환한다.
3. 전환 시에는 상대비교보다 `기준별 승자 선택 방식`을 우선 검토한다.

즉, 현재 시점의 추천은 아래와 같다.

- 단기: 현 구조 유지
- 중기: 기준별 승자 선택 방식으로 전환 검토
- 비추천: 후보가 많은 상태에서 상대비교 구조로 바로 가는 것

## 10. 결정 포인트

아래 질문에 대한 답이 정해지면 전환 여부를 결정하기 쉽다.

- compare를 "수치 비교 도구"로 볼 것인가?
- compare를 "빠른 결승전 도구"로 볼 것인가?
- 사용자가 2~3개 후보만 주로 넣는가?
- 결과 설명력보다 입력 속도가 더 중요한가?

현재 앱의 방향성을 기준으로 보면, compare는 장기적으로 "빠른 결승전 도구" 쪽이 더 어울린다.
