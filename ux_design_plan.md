# 🍱 "대신 골라줘!" UI/UX 기획 및 설계 문서

## A. 서비스 개요
- **서비스 한줄 소개**: 기준과 가중치를 통해 가장 합리적이고 납득 가능한 점심/저녁 메뉴를 찾아주는 스마트 결정 도우미.
- **해결하려는 문제**: "오늘 뭐 먹지?"라는 반복되는 고민 상황에서 생기는 감정 소모를 줄이고, 무작위 뽑기의 허무함이나 복잡한 지도/리뷰 검색의 피로도 대신 **내 취향이 빠르고 정확하게 반영된 결정**을 내릴 수 있도록 돕습니다.
- **타깃 사용자**: 매일 점심 메뉴를 고르기 힘든 직장인, 데이트/모임 메뉴를 고민하는 커플 및 소그룹, 야식/배달 메뉴를 고르는 1인 가구.
- **대표 사용 상황(Use Case)**:
  - (점심) 짧은 점심시간, 동료들과 '맛', '거리', '대기시간'을 적당히 타협하여 식당을 정할 때.
  - (저녁) 하루를 보상받고 싶은 저녁, '양'이나 '분위기', '가격'을 고려하여 배달/외식 메뉴를 고를 때.

## B. UX 전략
- **왜 '음식 특화형'이어야 하는가?**
  - 범용적인 의사결정 매트릭스 툴은 기준부터 점수까지 일일이 입력해야 해서 딱딱하고 번거롭습니다. 음식 선택에 특화된 친숙한 기준(맛, 거리, 가격 등)과 상황에 맞는 워딩을 미리 깔아두어 사용자의 **입력 피로도를 획기적으로 낮춰야** 합니다.
- **왜 '단계형(Wizard) UX'가 적합한가?**
  - 모바일 화면에서 기준, 후보, 가중치, 점수를 한 번에 입력하게 하면 인지적 과부하(Cognitive Overload)를 겪어 이탈합니다. 단계를 쪼개어 한 화면에 하나의 행동만 제시함으로써 폼 작성의 심리적 허들을 낮추고 마치 게임처럼 몰입하도록 유도합니다.
- **귀찮지 않게 만드는 핵심 UX 원칙**
  1. **사고의 중단 방지**: "가격이 낮을수록 좋으니 숫자를 어떻게 주지?" 같은 혼란을 방지하기 위해 모든 질문을 직관적인 **'만족도 중심의 긍정어'**로 치환합니다 (예: 거리가 가깝나요? 가성비가 좋은가요?).
  2. **기본값(Default) 적극 활용**: 사용자가 아무것도 조정하지 않고 "다음" 버튼만 눌러도 적당한 타협안이 나오도록 초기 균등 배포 상태(각 25% 등)와 기본 평점(3점)을 제공합니다.
- **가장 중요한 UX 포인트 3가지**
  - 퍼센티지 총합 100%가 자동 보정되는 슬라이더 디자인.
  - 리스트를 한눈에 비교해 피로도를 줄이는 '기준(Criteria) 중심 뷰'의 별점 평가.
  - 1위 결과를 수치적으로 설득하고 재미를 주는 결과 피드백(Justification) 연출.

## C. 사용자 흐름 (User Flow)

**Step 1. 점심 / 저녁 선택 (Mode Selection)**
- **행동**: '점심' 또는 '저녁' 중 하나의 거대한 카드 단위 탭.
- **화면 목적**: 사용 상황을 선언하고 화면의 분위기를 조성함과 동시에 테마(밝음/어두움)를 세팅.
- **이탈 방지**: 앱 실행 직후 고민 없이 누를 수 있는 가장 가벼운 첫 상호작용. 동작 즉시 스크롤/화면 전환.

**Step 2. 기준 가중치 설정 (Weight Configuration)**
- **행동**: 기본 항목(맛, 가격, 거리, 대기시간)의 슬라이더 조절. ('+ 기준 추가'로 칩 선택 가능)
- **화면 목적**: "이번 식사에서 가장 중요한 게 뭐야?"를 묻고 우선순위를 시각화.
- **이탈 방지**: 조작 없이도 균일하게 할당된 디폴트 값 제공. 슬라이더 드래그 시 Haptic(진동) 피드백 제공.

**Step 3. 후보 음식점 입력 (Candidate Input)**
- **행동**: 텍스트 입력창에 후보(식당명/메뉴명) 입력 후 엔터를 치면 태그(Chip) 형태로 전환되어 나열됨.
- **화면 목적**: 고민 중인 타겟들을 명확히 묶어두기.
- **이탈 방지**: 최소 2개만 입력하면 바로 입력 폼 하단의 [다음] 버튼 넛지 활성화.

**Step 4. 후보별 점수 평가 (Scoring)**
- **행동**: 각 후보에 대해 만족도 1~5점(또는 표정 이모지) 탭.
- **화면 목적**: 최종 계산 엔진을 돌리기 위한 기초 데이터 직관적 수집.
- **이탈 방지**: 식당 하나씩 띄워두고 평가하게 하면 피로도가 높습니다. 기준 하나를 띄우고 아래에 여러 식당을 리스트업 하여 "맛은 어디가 젤 낫지?" 하며 일괄 비교 터치하게 유도합니다.

**Step 5. 결과 확인 (Result & Justification)**
- **행동**: 대망의 1위 결과를 확인하고, 납득하거나 다시 고르기 액션 수행.
- **화면 목적**: 시각적 쾌감과 이성적 설득력(왜 이게 1위인지) 동시 제공.
- **다음 조건**: [다시 하기] 또는 [가중치만 살짝 바꿀래] 탭.

## D. 화면 구조 (Screen Structure) & 와이어프레임 묘사

- **[Screen 1] Home (상황 선택)**
  - **상단**: 로고 타입 "대신 골라줘!", 서브타이틀 "오늘의 식사는 무엇인가요?"
  - **본문**: 화면 전체 2분할 버튼. (상단/좌측: 🌞 옐로 톤의 '점심' / 하단/우측: 🌙 네이비 톤의 '저녁')
  - **하단**: 단순함을 위해 비워두거나 작은 안내 영역.

- **[Screen 2] Criteria (기준과 가중치)**
  - **상단**: 진행 인디케이터 (1/4), "이번 식사에서 제일 중요한 건 뭐예요?"
  - **본문**: 4개의 연결 슬라이더 리스트. 우측에 동기화되는 % 노출.
  - **하단**: 남은 여유 %를 보여주는 게이지 바 + 플로팅 된 [다음] 대형 버튼.

- **[Screen 3] Candidates (후보)**
  - **상단**: 진행 인디케이터 (2/4), "어디어디 고민 중이신가요?"
  - **본문**: 중앙 대형 인풋창. 입력 시 화면 중앙에 해시태그 뭉치처럼 [🥘 김치찌개 x] 칩 뷰 생성.
  - **하단**: "2개 입력됨" 문구와 함께 활성화되는 [다음 ➔] 버튼.

- **[Screen 4] Scoring (점수 평가)**
  - **상단**: 진행 인디케이터 (3/4), "기준별로 각 음식점을 평가해볼까요?"
  - **본문 와이어프레임 예시**:
    ```text
    [ 맛 ]  [📍가격 만족도]  [ 거리 ]  [ 대기 ]   ➔
    -------------------------------------------
     👇 가성비가 훌륭할수록 높은 점수(😍)를 주세요!
    
     🥘 A식당 (김치찌개)
     😡  😕  [ 😐 ]  🙂  😍 
    
     🍝 B식당 (돈까스)
     😡  [ 😕 ]  😐  🙂  😍 
    -------------------------------------------
    ```
  - **하단**: 평가가 전부 끝나면 팝아웃되는 [🔥 운명의 결과 보기] 버튼.

- **[Screen 5] Result (결과)**
  - **상단**: "빰빠라밤! 오늘의 승자는..." (드럼롤 텍스트 등장)
  - **본문**:
    - 거대한 1위 카드: 🥇 **김치찌개**
    - 설득형 문구 박스: "사용자님이 가장 중시하신 [가격(40%)] 부문에서 높은 점수를 차지해 최종 1위로 선정되었습니다."
    - 비교 차트 영역: 1~3위의 막대 그래프 (색상 블록으로 쪼개진 누적 차트로 구성 타당성 증명)
  - **하단**: [가중치만 수정해서 다시 랭킹 보기] / [완전히 자비 없이 초기화] 듀얼 버튼.

## E. UI 컴포넌트 제안

1. **Fluid Sliders (상호 연결된 액체형 슬라이더 페어)**
   - **적합성**: 수리적 고민 제거. 하나를 10% 올리면 나머지 슬라이더들이 스멀스멀 내려와 총합 100%를 자동 방어해 주는 동적 시각 피드백 기능 탑재.
2. **Emoji Selector (이모지 5단계 버튼)**
   - **적합성**: 딱딱한 1~5별점 대신 얼굴 표정을 두면 UI 감성이 올라갑니다. 무엇보다 "거리가 5점이면 먼 거야, 가까운 거야?" 같은 '역방향 문항 인지 에러'를 표정 하나로 차단할 수 있습니다.
3. **Chip Input Field**
   - **적합성**: 모바일 환경에서 텍스트 덩어리를 블록화(Chip) 하면, 화면에 꽉 찬 덩어리감을 주고 오타 시 삭제(X) 터치 반경도 넉넉해집니다.
4. **Stacked Bar Chart (누적 막대 차트)**
   - **적합성**: 결과 화면에서 1위와 2위를 비교할 때, 어떤 기준(색상)의 득점이 높아서 이겼는지 한눈에 설득할 수 있는 가장 훌륭한 시각 데이터 표상입니다.

## F. 마이크로 UX & 인터랙션 (감성 디테일)

- **Context Theme Switch**: 첫 화면에서 '저녁'을 고르면 배경색이 다크 모드로 부드럽게 페이드-인.
- **One-page Sequence**: '다음'을 누르면 화면 자체가 끊기고 바뀌는 게 아니라, 이전 카드가 밀려 올라가고 다음 카드가 스와이프업 하며 호흡을 잃지 않게 설계.
- **Shake Error 모션**: 똑같은 후보 레스토랑을 2번 썼을 때, 입력 박스가 좌우로 '도리도리' 흔들리며 위트 있게 방어.
- **Result Motion**: 버튼 클릭 시 잠시 로딩 스피너 대신 드럼롤 그래픽이 흔들리다가 카드가 팡!(Pop-out) 하고 터져 나오는 축하(Confetti) 모션.

## G. 제품 MVP (Minimum Viable Product) 스펙 결정

**✅ 1차 배포 필수 (Must Have)**
- 점심/저녁 토글형 테마 모드 전환 베이스.
- 4개 주요 기준(맛, 거리, 가격, 대기)이 기본 세팅된 100% 자동 분배 슬라이더.
- 태그 방식 후보 입력 및 이모티브(표정) 퀵 스코어 보드.
- 누적 막대 차트가 포함된 결과 화면과 가중치 기준 자동 설명 문구(AI 조합 텍스트).

**🤔 차후 개선 (Nice to Have) - 지금은 없어도 동작함**
- 추가 기준 커스터마이징 모달 (예: "다이어트 메뉴", "조용한 곳" 등 특수 칩 제공).
- "최근에 고민했던 식당 리스트" 또는 "자주 가는 단골집" 로컬 스토리지 불러오기 기능.
- 카카오/네이버지도 검색 퀵 외부 링크 삽입.

**❌ 초기 버전에선 빼야 할 기능 (Out of Scope)**
- 사용자별 로그인, 회원가입, 커뮤니티 공유 기능 (가벼운 유틸리티 성격이 퇴색됨).
- 실제 식당 API를 끌어와 후보를 자동 검색/완성 해주는 기능 (공수가 너무 크며, "회사 앞 구내식당" 같은 주관적 단어가 안 찾아질 리스크 다분. 유저 수기 타이핑이 우선).

## H. 예상 리스크 시나리오 및 방어 설계

⚠️ **리스크 1: "낮을수록 좋음(가격/거리)" 기준의 인지 부조화 오류**
- **문제**: "거리가 몇 점인가요?" ➔ 사용자는 '숫자가 낮아야 최고 점수인가?', '5점을 줘야 가까운 건가?' 3초간 머리가 멈춥니다.
- **개선안**: 기준 명칭을 "가성비 만족도", "가까움도(접근성)" 같이 **무조건 긍정형 만족도** 언어로 워딩 튜닝합니다. "싸서 만족스럽다=😍(5점)" 구조가 만들어져야 UX 정지를 막을 수 있습니다.

⚠️ **리스크 2: 수많은 문항 x 다수 후보 = "아 귀찮아" 평가 포기 현상**
- **문제**: 5개의 식당 후보, 4개의 기준이면 버튼을 총 20번 탭해야 하며 이는 치명적입니다.
- **개선안**: 식당 하나마다 4개 항목을 평가하지 않습니다. **"하나의 기준 아래 모든 식당 쪼라락 비교"** 뷰를 채택합니다. 모든 디폴트는 3점(보통 😐)으로 세팅하여, 사용자가 '김치찌개는 맛있으니까 5점, 돈까스는 짜니까 2점' 식으로 짚고 싶은 것만 클릭하고 바로 넘길 수 있게 체감 난이도를 낮춥니다.

⚠️ **리스크 3: 100% 자동 보정 슬라이더의 알고리즘 충돌(멘붕 가능성)**
- **문제**: 맛 슬라이더를 100으로 올려서 다른 게 모두 0이 되었을 때, 다른 0짜리 슬라이더를 또 올리면 알고리즘이 꼬이거나 미친 듯이 튕깁니다.
- **개선안**: 모든 항목이 0이 되지 않도록 각 항목당 최소 보장 치(예: 5%)를 두어 락(Lock)을 걸거나, 총합 스탯의 최대 한계치를 UI상 그릇 크기로 제약하여(예를 들어 최대 80%까지만 몰빵 가능하게 함) 오류를 유연하게 방어해야 합니다.
