# 🍱 "대신 골라줘!" UI 뼈대 수정안 (분기 흐름 추가)

기존 "점심/저녁 선택 ➔ 가중치 설정 ➔ 후보 ➔ 점수 ➔ 결과"로 이어지던 단일 선형 구조를 **최소한의 수정**으로 2가지 분기(투트랙) 버전으로 확장한 UI 명세서입니다.

---

## 1. 수정된 전체 화면 맵 (Screen Map)

`[Screen 1]`이 끝난 직후 **`[Screen 1.5] 분기 선택`** 화면이 개입하여 흐름을 두 갈래로 나눕니다.

```text
[Screen 1] Intro & Mode Select (점심/저녁)
      │
[Screen 1.5] Flow Select (분기 선택: 신규 추가)
      │
      ├─▶ (A. 빠르게 추천받기) 
      │       └─▶ [Screen A-1] 가벼운 취향 태그 선택 (신규)
      │       └─▶ [Screen A-2] 단일 메뉴/식당 랜덤 결과 (신규 적용)
      │
      └─▶ (B. 직접 비교하기)
              └─▶ [Screen B-1] 기존 Weight Wizard (가중치 설정)
              └─▶ [Screen B-2] 기존 Candidate Input (후보 입력)
              └─▶ [Screen B-3] 기존 Scoring Board (점수 평가 보드)
              └─▶ [Screen B-4] 기존 Result (비교 결과 차트)
```

---

## 2. 새로 추가되는 “어떻게 고를까?” 화면 명세

* **화면 이름**: Flow Select
* **화면 목적**: 사용자의 '현재 머리 아픈 정도'에 따라 최적의 추천 방식을 스스로 고르게 함
* **텍스트 와이어프레임**:
```text
[ < 뒤로가기 ]              
"어떻게 고를까요?"
(현재 고민되는 후보가 있다면 비교하기를 눌러주세요!)
--------------------------------------------------
 [ ⚡ 빠르게 추천받기 ]
   "머리 아파요, 그냥 알아서 하나만 딱 골라주세요."

 [ ⚖️ 직접 비교하기 ]
   "고민 중인 후보가 있어요. 가중치를 두고 승부를 봅니다."
--------------------------------------------------
```
* **이동 조건**: 둘 중 하나의 거대 카드를 탭하면 별도의 하단 [다음] 버튼 없이 즉시 각 루트의 첫 화면으로 이동합니다.

---

## 3. "빠르게 추천받기(Route A)" 흐름 제안 (단 2단계)

이 흐름은 최소한의 탭핑으로 즉각적인 결과를 얻는 것에 목표를 둡니다.

**[Screen A-1] 취향 태그 선택**
* **목적**: 완전한 랜덤이 아니라 '지금 끌리는 것'을 기반으로 필터링
* **UI 요소**: 다중 선택이 가능한 `Chip` 버튼 10여 개 나열 (Ex: 국물, 면, 고기, 매콤한, 헤비한, 밥, 샐러드 등)
* **이동 조건**: 최소 1개 이상 태그 선택 후 하단 [결과 보기 ➔] 활성화

**[Screen A-2] 단일 랜덤 픽 결과**
* **목적**: 선택한 취향 태그에 부합하는 메뉴 **오직 1개**를 강력하게 제시
* **UI 요소**: 화려한 `WinnerCard` 안에 메뉴 1개 노출 (Ex: 🥇 **얼큰한 짬뽕**)
* **재진입 인터랙션**: 하단에 `[마음에 안 들어, 다시 돌려!]` 버튼과 `[이럴 바엔 내가 직접 비교할래 (B루트로 이동)]` 보조 버튼 제공.

---

## 4. "직접 비교하기(Route B)" 유지 정책

**기존 UI/UX 구조를 100% 동일하게 유지**합니다.
* 가중치 설정, 태그 입력, 표정 탭 평가, 컴포넌트(슬라이더 연동, 막대 차트 연출) 등 기존 명세서에 정의된 것을 그대로 사용합니다.
* 개발 관점에서도 `currentStep` 1, 2, 3으로 관리하던 것을 `stepB-1`, `stepB-2` 같은 라우팅(상태) 뎁스만 하나 더 내려주면 되기 때문에 **내부 로직을 전혀 뜯어고칠 필요가 없습니다.**

---

## 5. 기존 컴포넌트 구조의 분해 및 재사용 설계

* **♻️ 그대로 재사용 가능한 것 (100% 활용)**
  * `AppShell`, `StepHeader`, `BottomActionBar`, `PrimaryButton`, `HelperText` 등 뼈대 컨테이너 및 헤더/푸터 전부.
  * `WinnerCard` (직접 비교 결과창 1위에 쓰던 레이아웃을 '빠르게 추천받기' 결과창에서도 글자만 바꿔 그대로 렌더링).

* **🛠️ 조금 수정할 것 (Props 확장)**
  * `ModeThemeCard` (도입부 점심/저녁 카드): 기존엔 아이콘과 텍스트가 고정이었으나, 이를 **[빠르게 추천 / 직접 비교]** 분기 화면에서도 재사용할 수 있도록 `icon`, `title`, `description`을 프롭스(Props)로 받아 렌더링하도록 껍데기를 추상화(리팩토링)합니다.
  * `ProgressBar`: 루트가 나뉨에 따라 분모 값(`totalSteps`)이 유동적으로 변해야 합니다. (A루트: 2스텝, B루트: 4스텝)

* **✨ 새로 추가할 것 (신규 컴포넌트)**
  * `TasteTagSelector`: A루트용 (클릭 시 토글되는 필터 칩 묶음)
  * A루트 전용 목업 데이터 리스트: (로컬 JSON 형태로 메뉴 DB 구성을 프론트 코드 내에 들고 있어야 당장 뽑기를 할 수 있음)

## 6. MVP 기준에서 너무 복잡해지지 않는지 점검 (Complexity Check)

* **결론: "복잡해지지 않으며, 오히려 MVP의 완성도가 극명히 올라가는 기획"**입니다.
* 사용자가 "후보가 없는데 앱을 켠 상황" (B루트로 진입 불가능한 유저)을 완벽히 방어(Bouncing Rate 하락)할 수 있습니다.
* 프론트엔드 작업 공수는 **[A-1 취향 필터링 화면] 하나의 UI만 더 그리면 되는 수준**입니다. (결과 카드는 재사용, 컴포넌트는 대부분 공통 블록이기에 로직만 간단히 덧붙이면 됨).
* 백엔드 API가 없는 MVP 버전이기에, 프론트 내부에 `const menuDB = [{name:'짜장면', tags:['면','중식']}, ...]` 형태의 간단한 하드코딩 배열만 두어도 충분히 훌륭하게 작동합니다.
