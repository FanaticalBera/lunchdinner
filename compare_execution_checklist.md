# LunchDinner 비교하기 개선 작업 체크리스트

이 문서는 `compare_review_report.md` 개정판을 실제 구현 작업으로 옮기기 위한 실행용 체크리스트다.
작업은 작은 단위로 끊고, 각 단계마다 `typecheck + test + 핵심 흐름 확인` 후 다음 단계로 진행한다.

## 공통 원칙

- 한 번에 구조를 다 바꾸지 않는다.
- 도메인 타입 변경이 UI 변경보다 먼저다.
- compare 핵심 흐름을 깨지 않는 범위에서 단계적으로 개선한다.
- 지도 연동은 이번 체크리스트 범위에서 제외한다.
- 각 단계 종료 후 `npm run typecheck`와 `npm run test`를 우선 실행한다.

## 진행 단계

### 1) 결과 데이터 구조 확장

- [ ] `src/domain/types.ts`의 `Result` 타입 확장
- [ ] ranking 항목에 기준별 점수 정보 구조 추가
- [ ] 필요 시 총점 외 보조 필드 추가 (`scoreByKey`, `weightedScoreByKey`, `scoreGap`, `isTie` 등)
- [ ] 새 타입 구조를 기준으로 화면 컴포넌트 타입 오류 정리

완료 기준
- [ ] `Result` 타입만 읽어도 결과 화면에 필요한 데이터가 무엇인지 드러난다.
- [ ] 결과 설명용 데이터를 UI가 별도 계산 없이 받을 수 있다.

### 2) `buildCompareResult()` 확장

- [ ] `src/domain/recommendation.ts`에서 기준별 원점수 계산 분리
- [ ] 기준별 가중 반영값 계산 추가
- [ ] 1위와 2위 점수 차이 계산 추가
- [ ] 동점 여부 및 동점 그룹 정보 계산 추가
- [ ] 변경된 결과 구조에 맞춰 반환값 정리

완료 기준
- [ ] `buildCompareResult()`만으로 결과 설명에 필요한 도메인 데이터가 모두 나온다.
- [ ] 결과 화면에서 추가 계산 로직이 최소화된다.

### 3) compare 도메인 테스트 보강

- [ ] `src/domain/recommendation.test.ts`에 결과 구조 확장 케이스 추가
- [ ] 기준별 점수 합산 검증 테스트 추가
- [ ] 1위와 2위 점수 차이 검증 테스트 추가
- [ ] 동점 케이스 상세 검증 추가
- [ ] 후보 2개 / 3개 / 4개 케이스 최소 샘플 확보

완료 기준
- [ ] 결과 구조 변경이 테스트로 고정된다.
- [ ] 동점/근소한 차이/일반 승리 케이스가 모두 검증된다.

### 4) 결과 화면 설명력 강화

- [ ] `src/screens/ResultScreen.tsx`에 기준별 breakdown 영역 추가
- [ ] 1위 vs 2위 비교 영역 추가
- [ ] 동점 시 문구와 다음 액션 안내 개선
- [ ] 근소한 승부일 때 요약 문구 차별화
- [ ] 상위 3개 목록 표시 방식을 새 데이터 구조에 맞게 정리

완료 기준
- [ ] 사용자가 "왜 1위인지"를 화면만 보고 이해할 수 있다.
- [ ] 동점과 일반 승리 화면이 같은 문구로 뭉개지지 않는다.

### 5) 후보 수 상한 제한

- [ ] `src/screens/CandidateInputScreen.tsx`에 최대 후보 수 상한 추가
- [ ] 권장 후보 수 (`2~3개`) 안내 문구 추가
- [ ] 상한 도달 시 입력 버튼/엔터 동작 차단
- [ ] 상한 초과 시 명확한 안내 메시지 표시
- [ ] 기존 상태에 후보가 많을 경우 방어 동작 확인

권장 기본값
- [ ] 최대 후보 수: 4개

완료 기준
- [ ] 사용자가 무제한 후보를 넣어 compare를 과도하게 무겁게 만들 수 없다.
- [ ] 후보 입력 단계에서 compare의 사용 범위가 명확히 드러난다.

### 6) 점수평가 화면 단기 보강

- [ ] `src/screens/ScoringBoardScreen.tsx`에 탭별 진행 상태 표시 추가
- [ ] 현재 기준에서 미입력 후보 강조 표시 추가
- [ ] 각 기준 의미 설명 문구 추가
- [ ] 1~5점 의미를 짧은 텍스트로 보강
- [ ] 후보 수가 4개일 때도 모바일 화면 가독성 확인

완료 기준
- [ ] 현재 구조를 유지하더라도 사용자가 덜 멈추고 입력할 수 있다.
- [ ] 점수 기준이 지금보다 명확해진다.

### 7) 점수평가 재설계안 탐색

- [ ] 기준별 승자 선택 방식의 상태 구조 초안 작성
- [ ] 현재 `weights + scores` 구조와 충돌 지점 정리
- [ ] `절대점수 x 가중치` 대비 `기준별 승리 x 가중치` 계산 규칙 비교
- [ ] UI 시안 수준으로 질문형 compare 흐름 정리
- [ ] 재설계 전환 여부를 결정하기 위한 장단점 표 정리

완료 기준
- [ ] 재설계를 할지, 현 구조를 유지 보강할지 근거가 문서화된다.
- [ ] 실제 구현 전에 상태/도메인 변경 범위가 명확해진다.

### 8) 후보 입력 품질 개선

- [ ] 공백/표기 차이를 고려한 중복 입력 방지
- [ ] 후보 수정 기능 추가 여부 결정 및 반영
- [ ] 예시 후보 칩 또는 힌트 문구 추가
- [ ] 삭제 후 재입력 흐름 불편 여부 점검

완료 기준
- [ ] 후보 입력 단계에서 사소한 입력 품질 문제로 UX가 흔들리지 않는다.

### 9) 거리 정보 보조화 설계

- [ ] 지도 없이 거리 기준을 다루는 방식 초안 작성
- [ ] 후보별 `가깝다 / 보통 / 멀다` 입력 방식 검토
- [ ] 현재 점수평가 구조와 결합 방식 정리
- [ ] 위치 기반 보조정보를 나중에 붙일 수 있는 확장 포인트 기록

완료 기준
- [ ] 지도 없이도 `distance` 기준을 더 설득력 있게 다룰 수 있는 방향이 정리된다.
